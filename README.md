# Arkme Demo

这是一个用于候选人笔试的移动端前端 Demo。候选人需要克隆本项目后，在 Codex 客户端中根据给定需求继续迭代。完成后将内容推到你自己的 GitHub 中，并把项目链接发给我们。

v0.1.4实现“聊天页顶部安排确认弹窗”功能： 
目标： 当 AI 从聊天内容中识别出待办/安排后，不再要求用户打开安排界面确认，而是在当前聊天界面顶部弹出一个确认卡片。 
功能： 
1. 在聊天页面顶部显示一个浮层/弹窗卡片。 
2. 卡片内容包括： - 识别出的安排标题 - 时间 - 地点（如果有） - 来源消息摘要 - 置信度或“AI 已识别到安排” 
3. 卡片按钮： - 确认添加 - 编辑 - 忽略 
4. 点击“确认添加”后，将该安排写入现有 todo/arrangement 存储。 
5. 点击“编辑”后，允许用户修改标题、时间、地点、备注后再保存。 
6. 点击“忽略”后，关闭卡片，并记录该识别结果已处理，避免重复弹出。 
7. 如果短时间内识别到多个安排，用队列方式逐个显示，或折叠成“发现 3 个安排”。 
8. 弹窗不要遮挡聊天输入框，建议固定在聊天页面顶部。 
v0.1.4.1优化了弹窗内容，缩小了弹窗的面积
v0.1.5实现“已确认安排的后续聊天自动补全更新”功能： 
当用户已经确认一个安排事项后，如果后续聊天中继续出现该事项的时间、地点、备注、参与人等信息，AI 应该能识别这些信息属于已有待办，并自动更新该待办。 
功能： 
1.已确认的安排事项保存后，继续监听后续聊天消息。 
2.AI 识别新消息时，不仅判断是否是新安排，还要判断是否是在补充已有安排。 
3.如果新消息和已有待办语义相关，例如： - “时间改到明天下午三点” - “地点在图书馆三楼” - “改成线上腾讯会议” - “带上身份证” 应更新原待办，而不是新建待办。
4.对已有待办更新无需在聊天顶部弹出确认卡片，自动调整时间，地点在安排界面进行更新，当用户打开安排界面再进行确定。 
5.如果置信度高，可以直接更新，但建议保留可撤销提示。 
6.避免重复更新同一条消息。
7.保持原有“新建安排确认弹窗”不变。

修复后续聊天补全时错误匹配待办的问题：

问题：
“待会儿下班了一起去喝一杯吗”应该生成“喝一杯/喝酒”的安排。
后续“8点怎么说”“可以”应该补全到喝酒安排，而不是补全到去网吧安排。

要求：
1. AI 更新已有待办时，不能只按最近待办匹配。
2. 必须结合以下因素打分：
- sourceMessage 是否属于同一聊天会话
- 前后消息是否围绕同一个话题
- 待办标题/地点/备注与当前消息语义是否相关
- 参与人是否一致
- 时间距离是否合理
- 是否是用户对上一条问题的回答
3. 如果当前消息是短句，例如：
- “可以”
- “8点”
- “行”
- “那就这样”
必须优先绑定到最近一轮未闭合的聊天意图，而不是随便匹配已有待办。
4. 新增 activeConversationIntent：
用来记录最近正在讨论的安排主题。
5. 只有当 confidence >= 0.75 才允许自动更新。
6. 如果有多个候选事项分数接近，必须弹出确认卡片让用户选择：
- 更新“喝一杯”
- 更新“去网吧”
- 新建安排
- 忽略
7. 修复该案例：
用户A：“待会儿下班了一起去喝一杯吗”
我：“可以”
我：“我要去”
我：“几点”
用户A：“8点怎么说”
我：“可以”
结果应该是：
- 事项：喝一杯
- 时间：晚上 8 点
不应该更新“去网吧”。

v0.1.5实现“用户自定义隐喻/暗语识别”功能： 
目标： 允许用户把某些抽象表达、符号、暗语、隐喻词绑定到真实事项，例如： “老地方” = “学校门口奶茶店” “开黑” = “打游戏” “补蓝” = “喝咖啡/休息” 要求： 
1.新增用户自定义 aliasMap。 
2. AI 识别聊天内容前，先进行 alias 解释。 
3. 如果消息包含用户定义的 alias，则把 alias meaning 一起传给 AI。
 4. AI 返回时保留原文和解释后的含义。 
5. 不能对未定义的隐喻强行猜测。
 6. 如果 AI 认为某句话可能是隐喻，但没有用户定义，弹出确认： - “这句话是不是代表某个安排？” - 用户可选择：绑定为新暗语 / 忽略 7. 支持用户后续修改或删除暗语。 8. 暗语只在当前用户或当前聊天上下文中生效，避免全局误判。
